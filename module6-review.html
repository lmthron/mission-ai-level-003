AI System Design Patterns: Architecting Intelligent Systems

This guide covers the essential concepts from Module 6 to help you prepare for the checkpoint quiz. Focus on understanding the four foundational patterns, state management strategies, context window challenges, and pattern selection frameworks.

Why Design Patterns Matter for AI

What Are Design Patterns?

Design patterns are reusable solutions to common problems. For AI systems, patterns solve challenges like:

  • How to chain multiple AI calls together
  • When to use different models for different tasks
  • How to verify and improve AI outputs
  • How to manage conversation state
  • How to handle context window limits

Traditional Software

  • ‚Ä¢ Deterministic: Same input always produces same output
  • ‚Ä¢ Predictable behavior
  • ‚Ä¢ Clear if/else paths

AI Systems

  • ‚Ä¢ Probabilistic: Same input might produce different outputs
  • ‚Ä¢ Unpredictable behavior
  • ‚Ä¢ Need verification and validation

‚ö†Ô∏è Key Insight

AI systems are probabilistic, not deterministic. This fundamental difference, combined with statelessness, context limits, and cost per call, creates challenges that design patterns help solve through proven architectural solutions.

The Four Foundational Patterns

1. Pipeline

  • ‚Ä¢ Sequential processing
  • ‚Ä¢ Output feeds next input
  • ‚Ä¢ Like assembly line
  • ‚Ä¢ Order matters

2. Router

  • ‚Ä¢ Classifies input first
  • ‚Ä¢ Routes to specialists
  • ‚Ä¢ Like switchboard
  • ‚Ä¢ Different handlers

3. Evaluator

  • ‚Ä¢ Verifies AI output
  • ‚Ä¢ Pass/fail decision
  • ‚Ä¢ Like quality control
  • ‚Ä¢ Defined criteria

4. Feedback Loop

  • ‚Ä¢ Self-critique output
  • ‚Ä¢ Iterate until good
  • ‚Ä¢ Like self-editing
  • ‚Ä¢ Multiple attempts
Pipeline Pattern

What Is Pipeline?

Pipeline means the output of one AI call feeds into the input of the next AI call sequentially. Think assembly line: each station does one specific job and passes the result to the next station.

‚úÖ Use Pipeline When:

  • ‚Ä¢ Task breaks into sequential steps
  • ‚Ä¢ Each step has clear input/output
  • ‚Ä¢ Order matters (can't skip steps)
  • ‚Ä¢ Different prompts per stage
  • ‚Ä¢ Deterministic flow needed

‚ùå Pipeline Limitations:

  • ‚Ä¢ No flexibility (all steps every time)
  • ‚Ä¢ Can't skip or reorder dynamically
  • ‚Ä¢ If one step fails, entire pipeline fails
  • ‚Ä¢ Every request pays full pipeline cost
  • ‚Ä¢ Not for dynamic decision-making

Pipeline Examples

  • Content generation: Research ‚Üí Outline ‚Üí Draft ‚Üí Edit ‚Üí Format
  • Data processing: Extract ‚Üí Transform ‚Üí Validate ‚Üí Summarize
  • Document analysis: Parse ‚Üí Extract ‚Üí Categorize ‚Üí Report
Router Pattern

What Is Router?

Router means an initial AI call classifies the input and directs it to the most appropriate specialized handler. Think switchboard operator: determines which department should handle the call, then routes accordingly.

‚úÖ Use Router When:

  • ‚Ä¢ Clear request categories exist
  • ‚Ä¢ Different handling per category
  • ‚Ä¢ Specialized expertise needed
  • ‚Ä¢ Cost optimization (simple ‚Üí cheap model)
  • ‚Ä¢ Scalable category additions

‚ùå Router Limitations:

  • ‚Ä¢ Wrong classification = wrong handler = bad result
  • ‚Ä¢ Router is single point of failure
  • ‚Ä¢ Must handle ambiguous cases
  • ‚Ä¢ Maintenance of multiple handlers
  • ‚Ä¢ Not for queries spanning categories

‚ö†Ô∏è Critical: Router Accuracy

Router accuracy is critical because wrong classification sends requests to the wrong specialized handler, producing poor results. The router itself is a single point of failure. Architects must plan for ambiguous cases and ensure the classifier is reliable.

Evaluator Pattern

What Is Evaluator?

Evaluator means using AI to verify, score, or critique other AI outputs before finalizing. Think quality control: AI generates output, another AI (or same AI with different prompt) evaluates it against defined criteria.

‚úÖ Use Evaluator When:

  • ‚Ä¢ Quality matters more than speed
  • ‚Ä¢ High-stakes outputs (customer comms)
  • ‚Ä¢ Objective criteria can be defined
  • ‚Ä¢ Cost of errors is high
  • ‚Ä¢ Consistency requirements exist

‚ùå Evaluator Limitations:

  • ‚Ä¢ Extra AI calls (2-3x base cost)
  • ‚Ä¢ Adds latency to response
  • ‚Ä¢ Evaluator itself can be wrong
  • ‚Ä¢ Criteria must be well-defined
  • ‚Ä¢ Not for real-time needs
Feedback Loop Pattern

What Is Feedback Loop?

Feedback Loop means AI reviews its own output, identifies issues, and generates improved versions iteratively. Think self-editing: writer creates draft, reviews it, identifies problems, rewrites. Repeat until satisfied.

‚úÖ Use Feedback Loop When:

  • ‚Ä¢ First attempt rarely good enough
  • ‚Ä¢ Quality improves with iterations
  • ‚Ä¢ AI can evaluate its own output
  • ‚Ä¢ Time/budget allows 2-5 attempts
  • ‚Ä¢ Self-contained task

‚ùå Feedback Loop Limitations:

  • ‚Ä¢ Cost multiplier (3-5x base cost)
  • ‚Ä¢ Convergence not guaranteed
  • ‚Ä¢ Can get stuck in loops
  • ‚Ä¢ Diminishing returns after first iteration
  • ‚Ä¢ Not for budget-constrained tasks
Evaluator vs. Feedback Loop

Evaluator Pattern

  • ‚Ä¢ Like quality control
  • ‚Ä¢ Generate ‚Üí Evaluate ‚Üí Pass/Fail
  • ‚Ä¢ Binary decision (meets criteria or not)
  • ‚Ä¢ May regenerate if fails
  • ‚Ä¢ External criteria check

Feedback Loop Pattern

  • ‚Ä¢ Like self-editing
  • ‚Ä¢ Generate ‚Üí Self-Critique ‚Üí Improve ‚Üí Repeat
  • ‚Ä¢ Iterative refinement
  • ‚Ä¢ Always tries to improve
  • ‚Ä¢ Internal reflection

Key Distinction

Evaluator uses AI to check output and passes or fails it against external criteria. Feedback Loop has the AI iteratively self-critique and regenerate improvements. Both add cost but serve different purposes ‚Äî Evaluator for verification, Feedback Loop for refinement.

Combining Patterns for Complex Systems

Patterns Work Together

Real systems rarely use just one pattern. Sophisticated applications combine patterns to handle complexity. For example, an enterprise support system might use:

  • Router: Classify question type (technical, billing, account)
  • Pipeline: Each route has stages (retrieve docs ‚Üí generate ‚Üí format)
  • Evaluator: Check response quality before sending
  • Feedback Loop: If evaluation fails, regenerate with improvements

Pattern Composition Guidelines

  1. Start simple: Begin with a single pattern
  2. Add complexity only when needed: Solve specific problems as they arise
  3. Prove value before adding features: Each pattern adds cost and complexity
  4. Don't over-engineer: Simplest solution that works is best

‚ö†Ô∏è Key Principle

Begin with a single pattern and add complexity only when specific problems require it. Each pattern adds latency, cost, and maintenance burden. Complexity has real costs in development, maintenance, and debugging.

State Management and Memory

The Stateless Problem

Critical concept: AI models are stateless. They don't remember previous interactions. Each AI call is independent with no memory of past conversation, no awareness of previous requests ‚Äî each interaction starts fresh.

  • Users expect memory: "As I mentioned earlier..."
  • But AI won't provide it automatically
  • You must engineer memory into your system

Conversation History

  • ‚Ä¢ Send full history each request
  • ‚Ä¢ Simple to implement
  • ‚Ä¢ Perfect recall of recent chat
  • ‚Ä¢ Grows with every message
  • ‚Ä¢ Expensive over time

Sliding Window

  • ‚Ä¢ Keep only last N messages
  • ‚Ä¢ Prevents overflow
  • ‚Ä¢ Controls costs
  • ‚Ä¢ Loses older context
  • ‚Ä¢ Arbitrary cutoff

Summarization

  • ‚Ä¢ Summarize older messages
  • ‚Ä¢ Keep recent ones full
  • ‚Ä¢ Retains important context
  • ‚Ä¢ Summaries lose details
  • ‚Ä¢ Extra AI calls needed

RAG for Long-term Memory

  • ‚Ä¢ Store facts in vector database
  • ‚Ä¢ Retrieve when relevant
  • ‚Ä¢ Works across sessions
  • ‚Ä¢ Unlimited long-term memory
  • ‚Ä¢ Requires infrastructure

Structured Memory Slots

  • ‚Ä¢ Specific slots for info types
  • ‚Ä¢ user_goal, constraints, decisions
  • ‚Ä¢ Structured and efficient
  • ‚Ä¢ Only relevant info included
  • ‚Ä¢ Must design schema upfront

Memory Strategy Selection

  • Short conversations (<10 exchanges): Use conversation history ‚Äî simple and effective
  • Medium conversations (10-50 exchanges): Use sliding window or summarization ‚Äî manage context size
  • Long-running applications (days/weeks): Use RAG or structured slots ‚Äî persistent memory
  • Multi-session applications: Combine approaches ‚Äî slots for key facts, RAG for details
Context Window Management

What Is a Context Window?

Every AI model has a context window ‚Äî the maximum amount of text it can process at once. Everything counts toward the limit: instructions, conversation history, retrieved documents, user message, and AI response.

The Context Window Challenge

As context grows, problems emerge:

  • Performance degradation: Models perform worse with very long context
  • "Lost in the middle" problem: Models ignore middle sections of long contexts
  • Cost increases: You pay per token ‚Äî longer context = higher cost
  • Latency increases: More tokens = slower processing

‚ö†Ô∏è "Lost in the Middle" Problem

Models pay more attention to the beginning and end of long contexts, potentially missing middle content. Research shows that even within stated limits, AI models tend to focus on information at the start and end, potentially missing or underweighting what's in the middle. This is why strategic context management matters ‚Äî more isn't always better.

Context Management Strategies

  • Prioritize essential information: Must-haves first, nice-to-haves if space allows
  • Context compression: Summarize older portions, extract key facts
  • Dynamic context loading: Only include what's needed for current request
  • Context window routing: Route to model based on context needs (small ‚Üí cheaper model)
Decision Trees for Pattern Selection

Pattern Selection Questions

  1. Is the task naturally sequential? Yes ‚Üí Consider Pipeline Pattern
  2. Do you have distinct categories of requests? Yes ‚Üí Consider Router Pattern
  3. Is output quality verification critical? Yes ‚Üí Consider Evaluator Pattern
  4. Does quality improve through iteration? Yes ‚Üí Consider Feedback Loop Pattern

Multi-Pattern Decision Framework

For each use case, ask:

  • Need sequential processing? ‚Üí Add Pipeline
  • Have multiple request types? ‚Üí Add Router
  • Quality assurance critical? ‚Üí Add Evaluator
  • Iterative improvement helpful? ‚Üí Add Feedback Loop
  • Need conversation memory? ‚Üí Add State Management
  • Long context? ‚Üí Add Window Management
Diagramming and Documentation

Why Diagrams Matter for AI Systems

AI systems are different from traditional software: They have probabilistic outputs, require coordinating multiple AI calls, need explicit state management, and face context window constraints. Diagrams help teams:

  • Understand AI call sequences and dependencies
  • See where prompts are used in the flow
  • Identify state management needs and storage points
  • Plan for failure cases and error recovery
  • Communicate architecture decisions to stakeholders

Essential Diagram Types

  • Component Diagram: Major system components and relationships (AI models, databases, APIs)
  • Sequence Diagram: Step-by-step flow of requests (AI calls in order, decision points)
  • State Diagram: How conversation state changes over time (transitions and triggers)
