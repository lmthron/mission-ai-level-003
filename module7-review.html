This guide covers the essential concepts from Module 7 to help you prepare for the checkpoint quiz. Focus on understanding the prototype-production gap, security requirements, performance optimization, cost management, deployment architectures, and operational excellence.

Prototype (Proving Concepts)

  • ‚Ä¢ Minimal error handling
  • ‚Ä¢ Performance "good enough"

  • ‚Ä¢ Secrets in secure vault

Over 70% of AI projects fail to reach production because teams underestimate this gap. What works on your laptop doesn't automatically work for thousands of concurrent users handling sensitive data with compliance requirements.

Security Considerations

API Key and Credential Management

  • Never hardcode keys: Use environment variables or secret management systems
  • Secret managers: AWS Secrets Manager, Azure Key Vault, HashiCorp Vault
  • Rotate regularly: Every 90 days minimum
  • Separate environments: Different keys for dev/staging/production
  • Revoke immediately: If any key is compromised

Access Control (RBAC)

Role-Based Access Control defines who can do what:

  • Authentication: Who are you?
  • Authorization: What can you do?
  • Least privilege: Only grant access that's needed

Prompt Injection Protection

The risk: Malicious users try to override your instructions.

  • Input validation: Filter suspicious patterns, block system instruction keywords
  • Output filtering: Check for sensitive data exposure, prevent unauthorized actions
  • Guardrails: Define what AI can/cannot do, escalate risky requests to humans
Performance Optimization

Exact Match Cache

  • ‚Ä¢ Identical queries only
  • ‚Ä¢ Simple to implement
  • ‚Ä¢ 10-20% hit rate typical
  • ‚Ä¢ Limited utility

Semantic Cache

  • ‚Ä¢ Similar meanings match
  • ‚Ä¢ "Refund policy?" = "How do refunds work?"
  • ‚Ä¢ 40-60% hit rate
  • ‚Ä¢ Much more effective

Prompt Cache

  • ‚Ä¢ Provider caches prompts
  • ‚Ä¢ System instructions cached
  • ‚Ä¢ 90% input cost reduction
  • ‚Ä¢ Claude/OpenAI support

Other Optimization Techniques

  • Model routing: Match model to task complexity - simple tasks use cheaper models
  • Batching: Process multiple requests together for batch jobs (5-10x throughput)
  • Streaming: Send tokens as generated - feels faster even if total time is same
Cost Management at Scale

‚ö†Ô∏è Critical: Output Tokens Cost More

Output tokens cost 3-5x more than input tokens. Don't default to max_tokens=4096 for everything. Set appropriate limits: FAQ answers might need 100 tokens, detailed explanations 500, full analyses 2000.

Prompt Optimization

Biggest immediate impact on costs:

  • Eliminate verbosity: "Analyze feedback for: sentiment, concerns, features" vs. lengthy instructions
  • Place static content early: Enables prompt caching
  • Put user question last: Improves performance by up to 30% for long contexts

Model Cascading

Use cheaper models first, escalate only when needed:

  • Step 1: Try cheap/fast model first
  • Step 2: If confidence low, escalate to mid-tier model
  • Step 3: If still uncertain, use most capable model

Research shows 80% of queries can be handled by cheaper models. This can reduce costs by 70% without quality loss.

When to Consider Self-Hosting

Only consider self-hosting if:

  • Processing over 10 million requests per month
  • Data residency absolutely requires it
  • You have a dedicated ML ops team

Otherwise, start with API-based deployment.

Rollback and Failure Handling

Graceful Degradation

When AI fails, system still works:

  • Primary: AI-generated response
  • If AI fails ‚Üí Fallback: Template/cached response
  • If template fails ‚Üí Fallback: Error message with human escalation

Users get help even when primary systems are down.

Retry Logic

  • ‚Ä¢ First attempt fails ‚Üí Wait 1s, retry
  • ‚Ä¢ Second fails ‚Üí Wait 2s, retry
  • ‚Ä¢ Third fails ‚Üí Wait 4s, retry
  • ‚Ä¢ Fourth fails ‚Üí Use fallback
  • ‚Ä¢ Exponential backoff prevents hammering

Circuit Breaker

  • ‚Ä¢ Service returns many errors
  • ‚Ä¢ Circuit "opens" (stops calling)
  • ‚Ä¢ Waits for cooldown period
  • ‚Ä¢ Tries again after cooldown
  • ‚Ä¢ Prevents cascade failures
Version Control for Prompts

Why Prompts Need Version Control

Prompts are code that controls AI behavior. They need the same rigor as application code:

  • History: See what changed, identify what broke, roll back
  • Collaboration: Multiple editors without overwriting each other
  • Testing: Test before production, not after users discover problems

Semantic Versioning for Prompts

Use MAJOR.MINOR.PATCH format:

  • MAJOR (v1 ‚Üí v2): Complete restructure, breaking changes, different output format
  • MINOR (v1.0 ‚Üí v1.1): Add examples, refine instructions, improve quality
  • PATCH (v1.0.0 ‚Üí v1.0.1): Fix typos, clarify wording, small adjustments

Best Practices

  1. Store prompts outside code: In database or config file, update without deployment
  2. Document changes: What changed, why, impact, who, when
  3. Test before production: Evaluate on test dataset, A/B test with small user percentage
  4. Separate environments: Development (experimental), Staging (testing), Production (stable)
Testing and Quality Assurance

‚ö†Ô∏è Testing AI Is Different

Traditional software is deterministic (same input = same output). AI systems are probabilistic (same input might produce different outputs). This requires new testing approaches.

Test Levels

  • ‚Ä¢ Unit: Test individual pieces (embeddings, retrieval)
  • ‚Ä¢ Integration: Test complete workflows end-to-end
  • ‚Ä¢ Regression: Ensure changes don't break existing functionality
  • ‚Ä¢ A/B Testing: Compare versions with real users

Quality Metrics

  • ‚Ä¢ Accuracy: Are responses factually correct?
  • ‚Ä¢ Relevance: Do responses address the question?
  • ‚Ä¢ Consistency: Similar questions get similar answers?
  • ‚Ä¢ Safety: No harmful content or PII leakage?

Regression Testing for AI

  • Build test dataset: 50-100 representative queries with known good responses
  • Run after every change: Deploy new version, run full suite, compare scores
  • Flag degradations: Catch problems before users do
