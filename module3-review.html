Client-Server Architecture

Three Layers

  1. MCP Host (The Application): AI-powered application users interact with (Claude Desktop, ChatGPT, Cursor, VS Code). Manages multiple client connections.
  2. MCP Client (The Protocol Handler): Translator that sits between AI and external tools. Converts AI requests to MCP protocol messages, sends to servers, translates responses back. Built into or added to host application.
  3. MCP Server (The Tool Provider): Lightweight program that exposes specific capabilities (GitHub server, Slack server, database server). Provides tools, resources, and prompts to clients.

The Embassy Analogy

  • You: The AI application
  • Embassy staff: MCP Clients (they speak both languages)
  • Foreign officials: MCP Servers (they have the information you need)

You don't speak the foreign language directly. Embassy staff (MCP Client) translates your requests, gets information from officials (MCP Server), and translates it back to you.

The Three Core Capabilities

1. Tools (Actions AI Can Perform)

What they are: Actual functions that perform actions on external systems

Examples: create_support_ticket, search_database, send_email, update_calendar, commit_code

How they work: Each tool has defined input parameters, returns structured output, AI decides when to use based on user request

2. Resources (Data AI Can Access)

What they are: Information the server manages that AI can read

Examples: File contents, database records, API responses, document contents, configuration data

How they work: Typically read-only access, structured data AI can process, no action required

3. Prompts (Templates for Common Tasks)

What they are: Pre-defined prompt templates that guide AI for common workflows

Examples: "Analyze customer feedback," "Generate monthly report," "Code review checklist"

How they work: Server provides template structure, AI fills in with current data, makes complex workflows repeatable

Transport Mechanisms

stdio (Standard Input/Output)

  • ‚Ä¢ For local servers on your machine
  • ‚Ä¢ Lightweight and simple
  • ‚Ä¢ Development tools, file access
  • ‚Ä¢ Personal productivity
  • ‚Ä¢ Security through isolation

HTTP with SSE / Streamable HTTP

  • ‚Ä¢ For remote servers over network
  • ‚Ä¢ Shared resources, cloud services
  • ‚Ä¢ Team collaboration
  • ‚Ä¢ Enterprise deployments
  • ‚Ä¢ Centralized management

Decision Principle

Local servers use stdio, remote servers use HTTP.

The protocol is transport-agnostic - the package contents (MCP messages) are the same, only the delivery method changes.

Security Model

Core Principle: Nothing By Default

MCP follows a zero-trust approach where nothing is exposed or allowed unless explicitly configured.

  • No tools available unless you configure them
  • No data accessible unless you grant permission
  • No actions possible without approval settings
  • API keys stay on your infrastructure

This is fundamentally different from "open by default" systems. MCP starts locked down and you explicitly open what's needed.

Where API Keys Stay

Critical security feature: API keys stay on the MCP server, not exposed to AI providers.

  • Users authenticate to the MCP server, not to the AI platform
  • Your database credentials never get sent to Anthropic, OpenAI, or Google
  • They remain in your infrastructure, controlled by your security policies

Permission Levels

  • Tool-level: Read vs. write operations clearly separated, each tool explicitly configured
  • Resource access: Granular control over what data AI can access, filter by sensitivity
  • User approval: Configurable - can require approval before execution

Build vs. Buy MCP Servers

The Principle

Default to existing servers for standard integrations (80% of use cases).

Build custom only when:

  • The system is proprietary to your company
  • No adequate community or official server exists
  • The investment is justified by usage volume or strategic importance
  • You have resources to maintain it long-term

‚úÖ Use Existing Servers

  • ‚Ä¢ Slack, GitHub, Google Drive, databases
  • ‚Ä¢ Well-tested, maintained by vendors
  • ‚Ä¢ Security reviewed, production-ready
  • ‚Ä¢ Faster time to value
  • ‚Ä¢ Free and open source

‚úÖ Build Custom When:

  • ‚Ä¢ Internal proprietary systems

  • ‚Ä¢ Business logic specific to you
  • ‚Ä¢ Sensitive systems needing custom security

Hybrid Approach (Most Common)

Use standard servers for common tools (Slack, GitHub, databases). Build custom servers for proprietary systems (internal CRM, transaction systems). Balance velocity with control.
